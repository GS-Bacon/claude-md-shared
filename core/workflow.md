# Workflow (壁打ち・Plan mode・コミット)

## Plan mode は壁打ち専用

- Plan mode に入っても、**ユーザーから明示的な指示があるまで Plan ファイルを作成しない**
- まず壁打ち(対話ベースの検討・質問・整理)に徹する
- 「計画を作って」「プランに起こして」「書き込んで」「プランファイルに書いて」等の明示指示でプランファイル作成・ExitPlanMode に進む

## 壁打ち中の振る舞い

- ユーザーの案を鵜呑みにしない。問題点・トレードオフ・代替案を率直に提示する
- 適宜、反論・批判・別案・調査を挟む
- 質問・敵対的レビューを通じて客観的視点を入れる
- ユーザーが見落としていそうな前提・制約があれば指摘する

## 自己判断でやってよいこと

- **コミット・プッシュ**: 区切りのよいところで自己判断で行う(ユーザー確認は不要)。破壊的操作(force push、reset --hard、ブランチ削除等)は事前確認
- **ライブラリのインストール**: 必要なものは許可なくインストールしてよい
- **ブランチ運用**: 指示がなければブランチを切らず、メインブランチへ直接 commit してよい
- **作業の継続**: 確認を求めず最後まで完了させる(完了まで一気に走る)

## バグ修正は再現テスト先行

**再現テストなしの修正を禁止する**。

1. バグ報告を受けたら、まず**バグを再現する自動テストを書く**
2. テストが期待通り**失敗すること**を確認する(再現できている証拠)
3. **修正を実装する**
4. テストが**通ること**を確認する(修正できている証拠)
5. 再発防止のためテストはコミットに含めて残す

理由: 再現できない修正は「直ったつもり」を量産する。自動化されない再現手順は次回も同じ問題で時間を溶かす。

例外: UI 表示のような自動テストが現実的でない領域は、再現手順を文章で残し、最低限の手動確認を実施する。

## 作業ログ

プロジェクトに `WORKLOG.md` や `WORK_LOG.md` が存在する場合、コミット・プッシュのタイミングで追記する。
無いプロジェクトでは作業ログを作らない(prompt 駆動で十分なため)。

## メタワーク回避

以下のような「成果物が増えるだけで本質が進まない作業」は避ける:

- 仕様書の詳細化のための詳細化
- ドキュメント整理・圧縮のためのドキュメント整理
- 使わないレビュースキル・パターン集の作成
- コード移動だけのリファクタ(複雑さが減らないもの)

判定基準: 「これをやるとバグが減るか・機能が進むか・将来の開発が速くなるか」のいずれにも該当しなければやらない。
